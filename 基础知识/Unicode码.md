# Unicode码

Unicode 源于一个很简单的想法：将全世界所有的字符包含在一个集合里，计算机只要支持这一个字符集，就能显示所有的字符，再也不会有乱码了。

由于Unicode中编码的字符数量太多，所以采取了平面的分类方式，目前一共有17个平面（0~16），每个平面是256（2^8）*256= 65536个字符，所以理论上Unicode可以编码65535*17=1114095个字符（百万级别）。从图中可以看出，连续平面的结束字符编码和开始字符编码是连续的，例如0号平面的U+FFFF和1号平面的U+10000。同时这里的Unicode编码方式使用的是16进制表示，即U+FFFF代表使用两个字节十六进制来表示，最多的例如U+10FFFF使用三个字节十六进制来表示，如下图表示。

注意，以码点U+FFFF为例，Unicode的码点是一种概念上的处理，可以理解为方便人们记忆，而不代表在硬件层面真的使用两个字节就把其代表的字符进行处理。在硬件层面对Unicode字符进行处理就需要Unicode码转换格式（Unicode Transformation Format，UTF）来做。Unicode码点值有三种UTF：UTF-8，UTF-16，UTF-32

## UTF-32

UTF-32就是使用4个字节来存储字符，所有的unicode都能够直接表示，简单粗暴，但是存在一个问题，就是存储空间利用率太低。unicode最大码点空间利用率都不足2 / 3，这样会浪费很多存储空间，对于网络传输也是一种负担。4个字节的长度意味着可以容纳任何码点的表示，所以其码点表示和UTF-32编码是一致的。



| 码点        | **UTF-32**  |
| ----------- | ----------- |
| U+0041: A   | 00 00 00 41 |
| U+03C9: ω   | 00 00 03 C9 |
| U+20AC: €   | 00 00 20 AC |
| U+2C9B0: 𬦰 | 00 02 C9 B0 |



## UTF-16

UTF-16 编码介于 UTF-32 与 UTF-8 之间，同时**结合了定长和变长两种编码方法的特点**。它的编码规则很简单：**基本平面的字符占用 2 个字节（一个平面有**65536个字符，而两个字节2^16=65536正好可以完全编码完成**），辅助平面（除了基本平面）的字符占用 4 个字节。**

UTF-16的解码过程难点在于如何判断一个字符是两个字节编码还是四个字节编码。

这里有一个很巧妙的地方，在基本平面内，从 U+D800到 U+DFFF 是一个空段（第55296个字符到57343个字符，共有57343-55296+1=2048个字符），即这些码点不对应任何字符。因此，这个空段可以用来映射辅助平面的字符。U+D800到 U+DFFF的空段又平均分成两段：

![img](https://ChihungSan.github.io/picx-images-hosting/jichuzhishi/image.60us37ulij.webp)

现在的思路是这样的：剩下所有辅助平面的字符数共有（**2^8\*2^8）\*16 = 2^20个，所以要对其进行编码的话（从0开始）需要20位才行。所以需要将原有的**unicode码点（十六进制）-10000（十六进制）得到新的的码点值。这样新的码点值就可以使用20位比特进行编码。

剩余**2^20=**1,048,576些字符显然无法一一对应到U+D800到 U+DFFF的空段。如果将20位的新码点值分成10位的两部分，而10位所能映射的范围正好是2^10=1024,正好是U+D800到 U+DFFF空段得到一半。所以可以将一个字符用两部分来映射。首先将新码点值填充0到20位，再平均分为2段（每段10位），再前面补0到16位，这样就是2*2=4个字节。同时还要将处理完成的码点移动到空段位置。

具体处理如下：以U+1F602为例

1. 判断U+1F602超过U+FFFF，所以不在**基本平面，需要4字节进行编码。**
2. 0x1F602-0x10000=0xF602
3. 0xF602 = 1111 0110 0000 0010 (16位)
4. 填充为：0000 1111 0110 0000 0010（20位）
5. 分成两段各10位：0000111101和1000000010
6. 将两段分别填充到16位（两个字节）：0000000000111101和0000001000000010
7. 两段分别代表的十六进制为：0x3D和0x202
8. 将两段分别映射到高半区和低半区
9. 0x3D+0XD800=0xD83D
10. 0x202+0XDC00=0xDE02
11. 最终编码结果为：0xD8，0x3D，0xDE，0x02

## UTF-8

以一个字节为单位对Unicode进行编码（意思是编码的最小单位为一个字节，而不是只有一个字节的意思），是一种**可变长度**的字符编码，可以使用 1 - 4 个字节表示一个Unicode字符。UTF-8 是一个非常惊艳的编码方式，漂亮的实现了对 ASCII 码的向后兼容，以保证 Unicode 可以被大众接受。

对于UTF-8编码而言，要对17平面的1114095字符进行编码，其规则分为两条如下：

- 使用一个字节进行编码和ASCII 码完全相同，即只能编码0~127的字符，按道理讲，一个字节八位最多可以表示2^8=256个字符，但其第一位始终为0，故只能编码2^7=128个字符（0~127）。
- 使用N（N>1）个字符进行编码。第一个字节(左侧)的前 N 位都设为1，第 N + 1 位设为0，剩余的 N - 1 个字节的前两位都设位 10，剩下的二进制位则使用这个字符的 Unicode 码点的二进制从后（右，最后一个位）向前来填充，不够的地方补0。

如下图所示，对于第一个平面U+0000~U+FFF而言，其包含的字符可以是1，2，3字节进行编码。而剩下的16个平面都是在4字节编码的范围内。如“汉”字符的码点为：U+6C49，所在范围是800~FFFF，所以应该使用三个字节进行编码，其编码格式要求为：1110 ****  10** ****  10** **** 十六进制6C49转为二进制为：110110001001001 填充后得到UTF-8编码结果：11100110，10110001，1000 1001。红色0为不够补的。

![img](https://ChihungSan.github.io/picx-images-hosting/jichuzhishi/image.9ddhxlb9p9.webp)

UTF-8解码的过程也十分简单：如果一个字节的第一位是 0 ，则说明这个字节对应一个字符；如果一个字节的第一位1，那么连续有多少个 1，就表示该字符占用多少个字节。
